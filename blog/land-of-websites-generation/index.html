Land of Websites generation

Websites are made up of 3 things

1. HTML
2. JS
3. CSS

Thats it.

Land of websites generation

  1. Static Site generation

    Write plain sites, build using vanilla html, css and js. Was happening initially

    Blog sites, news sites Good for those

    Technic of optimisations like JS compression this helps to reduce the bytes downloaded from server

    Other thing that can be done is can use different CSS frameworks like bootstrap/balma/tailwind CSS which provides CSS classes which can be used simplify styling

    You can cache it CDN like Cloud-front / Fastly

    Common practices

    1. Put assets on CDN
    2. Remove render blocking resources

    These are good for Simple static sites like blogs, some informative sites like news etc , which does not have much interactions. There is no server communication, its plain request for a resource in the form of html.

    Initial era pages where mostly plain html and served by Server

    Dynamically rendered server side pages:

    Here basically we are generating static pages using dynamic data from databases/service etc and generating html/css/js at build time and serving it statically. Even though these re generating from data from database and services. These are still static pages and don’t need to be generated again and again. This basically created need of tempating engine which will help user to generate html from marrying html with the domain data. JSP/ASP and some templating engines like handlebars are the one.

    There can be 2 categories of these one

    1. No frequent update of the data - in this case we only need to render the page once and server can safely serve the request. For example blog post which do no change with time, user can add the blogs over the time. When user adds new blog we can build it again and generate the html and store it on CDN. One of the challenge with storing the content on CDN is how and when to invalidate the content. Generally CDN caches based on the asset path. Path of assets like images, js/css files is generally referenced from html. If we can generate the assets with some version names and then cache on CDN you don’t need to apply any invalidation strategy for these assets. Real question is how to invalidate html. Because there is no versioning associated with it.

      One way to do this is to whenever there is new build published you can make explicit invalidate request to CDN to invalidate some assets that can be done using 2 ways

      1. Using url path: Make invalidation request based on path pattern. AWS does this
      2. Using tags: When origin respond to CDN, origin server tags the resource for example all the blog resources are tagged using “blog” keyword. When new build is produced at build time we can make request to invalidate the cache based on the tag.
      3. Using some salt in cache key: We can configure CDN to include some salt in cache key. Changing the salt will change the cache key which will automatically stop using already cached content and start caching new content based on new cache key. We can update the salt using some function as well, for example there is CloudFront function supported by CloudFront which acts like hook we can use those hooks to Update the request/response headers which can be included in cache key. Everytime the build changes now we need to update this edge function to update this header value.
      4. If it is ok to serve the old content for some time even though the content is refreshed in the background then we can use cache-control headers sent from origin to control the lifespan of the Content.

      So net net there are 3 ways to update/invalidate CDN cache.

       1. Explicitly make invalidation request to CDN to invalidate the cached content either by using path or tag. This will immediately make fresh content available to user.
 2. Use salt in cache key and update the salt to stop using already cached content. This basically kind of same as versioning. This will also immediately make fresh content available to user.
 3. Use cache control headers: This basically based on TTL.

      This CDN invalidations strategies are applicable to any way of generation web pages

    2. Sometimes data changes but frequency is not that often: The case where html content depends on the underlying data. Because of that 2 things can happen,

      1. Do not cache on CDN at all.

    3. Data may always change due to personalisation
